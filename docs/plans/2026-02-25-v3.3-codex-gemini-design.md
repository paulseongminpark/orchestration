# v3.3 설계: Codex/Gemini CLI 통합 강화

> 작성일: 2026-02-25
> 버전: v3.3
> 이전 버전: v3.2 (리좀형 4팀 + SoT 확립)
> 상태: 구현 완료 (2026-02-25)

---

## 1. 설계 의도

### 1.1 문제 정의

v3.2에서 Codex CLI와 Gemini CLI는 analyze 팀(gemini-analyzer, codex-reviewer, ai-synthesizer) 내에서 "전체 코드베이스 분석"이라는 모호한 트리거로만 정의되어 있었다. 실제 사용 빈도는 극히 낮았고, 구체적인 활용 시나리오가 부재했다.

**핵심 문제 3가지:**

1. **Claude가 Claude를 리뷰하는 구조**: code-reviewer가 Claude Opus로 돌아가므로 같은 모델의 맹점을 공유. 외부 모델에 의한 교차 검증이 구현 체인에 없음.

2. **컨텍스트 낭비**: 매 세션 시작 시 시스템 파악(에이전트 24개, 스킬 11개, 훅 7개, STATE.md 등)에 50K~100K+ 토큰을 소비. Opus 1M 중 10~15%가 "읽기"에만 사용됨.

3. **CLI 자체 기능 미활용**: Codex의 instructions.md, config profiles, review 명령과 Gemini의 GEMINI.md, skills, hooks 시스템이 전부 비어있는 상태로 운영.

### 1.2 설계 원칙

- **Claude = 유일한 설계/결정권자**: Codex/Gemini는 사실 확인과 추출만 수행. 설계 권한 없음.
- **해석이 아니라 추출**: 외부 CLI에게 "분석해"가 아니라 "추출해". JSON 구조화 출력으로 신뢰성 확보.
- **Verify Barrier**: 모든 외부 CLI 출력은 Claude가 최종 검증 후 사용. `_meta` 블록으로 정합성 체크.
- **자원 효율**: Codex(5시간 롤링 제한) = 정밀 작업. Gemini(일일 rate limit, 1M 컨텍스트) = 벌크 작업.

## 2. 설계를 통해 얻는 이점

### 2.1 컨텍스트 절약 (가장 핵심)

| 작업 | 현재 (Claude 직접 읽기) | v3.3 (외부 추출) | 절약 |
|------|----------------------|-----------------|------|
| 시스템 인벤토리 파악 | ~50K 토큰 | ~5K 토큰 | 90% |
| 프로젝트 소스 구조 | ~80K 토큰 | ~10K 토큰 | 87% |
| 크로스 프로젝트 STATE | ~40K 토큰 | ~5K 토큰 | 87% |
| Git 히스토리 | ~20K 토큰 | ~3K 토큰 | 85% |
| 함수 시그니처 캐시 | 반복 읽기 | 1회 추출 후 재사용 | 반복 제거 |
| **세션당 총 절약** | **~190K** | **~23K** | **~88%** |

Opus 1M 컨텍스트 중 ~170K를 실제 작업(설계, 구현, 대화)용으로 되찾음.

### 2.2 코드 품질 향상

- 다른 모델(GPT-5.3, Gemini 3.1 Pro)이 Claude가 놓치는 버그/취약점을 포착
- 보안 다중 스캐너 원칙: 모델마다 학습한 취약점 패턴이 달라 커버리지 증가
- completeness check + adversarial verify 2단계 검증

### 2.3 tech-review 콘텐츠 품질 향상

- Perplexity 단일 소스 의존 → Gemini Google 검색으로 교차 검증
- 팩트체크(freshness, coverage gap), 포맷 QA(Smart Brevity, KO/EN 일치)
- 기존 문제(주제 겹침, 말투 불일치, 얕은 조사) 구조적 해결

### 2.4 CLI 자체 기능 활용

- Codex: instructions.md, config profiles, review 명령, prompts 템플릿
- Gemini: GEMINI.md 계층 구조, custom skills, --output-format json, --yolo 모드
- 각 CLI의 네이티브 기능을 활용해 프롬프트 토큰 절약 + 일관된 행동 보장

## 3. 상세 설계

### 3.1 CLI-side 설정

#### 3.1.1 Codex CLI

**`~/.codex/instructions.md`** — 글로벌 페르소나

```markdown
# Codex 글로벌 지침

## 역할
Claude Code 오케스트레이션 시스템의 정밀 검증 도구.
설계 권한 없음. 사실 확인과 구조화 추출만 수행.

## 출력 규칙
- JSON 형식 우선. 자유 텍스트 최소화.
- 심각도 명시: CRITICAL / HIGH / MEDIUM / LOW
- 파일 경로와 줄번호 포함
- 해석/추천/칭찬/배경설명 금지. 사실만 보고.

## reasoning 지침
- 추출 전: 대상 파일을 반드시 전체 읽기
- 추출 후: 각 필드가 원본에 존재하는지 자체 검증
- 검증 실패 시: _meta.skipped에 기록
- diff 리뷰 시: 변경된 줄의 전후 맥락 5줄을 반드시 확인

## _meta 블록 (모든 JSON 응답에 필수)
응답 끝에 반드시 포함:
- files_scanned: 스캔한 파일 수
- fields_extracted: 추출한 필드 수
- skipped: [{item, reason}] — 스킵한 항목과 이유
- duration_hint: 작업에 사용된 대략적 추론 단계 수
```

**`~/.codex/config.toml`** — 프로필 체계

```toml
model = "gpt-5.3-codex"
model_reasoning_effort = "xhigh"
personality = "pragmatic"

[projects.'\\?\C:\dev']
trust_level = "trusted"

[windows]
sandbox = "elevated"

# 코드 리뷰 (기존)
[profiles.review]
model = "gpt-5.3-codex"
model_reasoning_effort = "medium"
sandbox_permissions = ["disk-full-read-access"]

# 구조화 추출 (신규) — 컨텍스트 오프로딩용
[profiles.extract]
model = "gpt-5.3-codex"
model_reasoning_effort = "high"
sandbox_permissions = ["disk-full-read-access"]

# completeness 체크 (신규) — 빠른 검증용
[profiles.verify]
model = "gpt-5.3-codex"
model_reasoning_effort = "low"
sandbox_permissions = ["disk-full-read-access"]

[tui]
status_line = ["model-with-reasoning", "current-dir", "context-remaining", "context-used", "five-hour-limit", "weekly-limit", "context-window-size", "used-tokens"]
```

프로필별 용도:
- `review`: diff 기반 코드 리뷰 (reasoning medium — 속도 우선)
- `extract`: 파일에서 구조화 데이터 추출 (reasoning high — 정확도 우선)
- `verify`: 추출 결과 completeness 체크 (reasoning low — 빠른 확인)

**`~/.codex/prompts/`** — 재사용 프롬프트 템플릿

```
~/.codex/prompts/
├── extract-git-history.md    # Git 최근 N일 히스토리 구조화
├── review-diff.md            # uncommitted diff 리뷰
└── content-qa.md             # tech-review 콘텐츠 QA
```

#### 3.1.2 Gemini CLI

**`~/.gemini/GEMINI.md`** — 글로벌 페르소나 + 추출 강제

```markdown
# Gemini 글로벌 지침

## 역할
구조화된 데이터 추출기. 해석하지 않는다.
Claude Code 오케스트레이션 시스템의 벌크 읽기 도구.

## 절대 규칙
1. JSON 형식으로만 출력
2. 원본에서 있는 그대로만 추출 — 요약/해석/판단/추천 금지
3. 누락 필드는 null로 표기 (추측하여 채우기 금지)
4. 원본에 없는 정보 추가 금지
5. 모든 응답에 _meta 블록 필수

## _meta 블록
{
  "_meta": {
    "files_scanned": 수치,
    "fields_extracted": 수치,
    "skipped": [{"item": "파일명", "reason": "이유"}],
    "model": "gemini-3.1-pro-preview",
    "completeness": "파일목록 수 == 추출 수 일치 여부"
  }
}

## reasoning 지침
- 추출 전: 대상 디렉토리의 파일 목록을 먼저 나열
- 추출 중: 나열한 각 파일에서 지정된 필드만 추출
- 추출 후: 나열한 파일 수 == 추출한 항목 수인지 검증
- 불일치 시: _meta.skipped에 기록하고 completeness를 "INCOMPLETE"로 표기
```

**`~/.gemini/skills/`** — 커스텀 스킬 4종

아래 4개 스킬을 생성:

##### (1) system-scanner

위치: `~/.gemini/skills/system-scanner/SKILL.md`

목적: Claude Code 오케스트레이션 시스템의 에이전트/스킬/훅 메타데이터를 구조화 추출.

트리거: "시스템 스캔", "에이전트 목록", "인벤토리", "시스템 파악"

절차:
1. `~/.claude/agents/*.md` 파일 목록 확인
2. 각 파일에서 YAML frontmatter의 name, description, model, tools 추출
3. `~/.claude/skills/*.md` 파일에서 name, description 추출
4. `~/.claude/settings.json`에서 hooks 섹션 추출
5. JSON 출력 + _meta 블록

출력 스키마:
```json
{
  "agents": [{"name":"","description":"","model":"","tools":[]}],
  "skills": [{"name":"","description":""}],
  "hooks": [{"event":"","command":""}],
  "_meta": {}
}
```

##### (2) project-scanner

위치: `~/.gemini/skills/project-scanner/SKILL.md`

목적: 프로젝트 소스 코드의 구조를 추출 (파일트리, exports, types, routes).

트리거: "프로젝트 구조", "소스 스캔", "아키텍처 파악"

절차:
1. 프로젝트 루트에서 파일 트리 생성 (node_modules, .git, dist, build 제외)
2. 각 TS/TSX/JS 파일에서 export 목록 추출
3. TypeScript interface/type 정의 추출
4. import 의존성 그래프 생성
5. 라우트 구조 추출 (Next.js app/ 또는 pages/)
6. JSON 출력 + _meta 블록

출력 스키마:
```json
{
  "file_tree": {"path":"","children":[]},
  "exports": [{"file":"","exports":["ComponentName"]}],
  "types": [{"name":"","fields":["field: type"]}],
  "imports": [{"source":"A.tsx","imports_from":["B.tsx","C.tsx"]}],
  "routes": ["/", "/about"],
  "_meta": {}
}
```

##### (3) state-scanner

위치: `~/.gemini/skills/state-scanner/SKILL.md`

목적: 복수 프로젝트의 STATE.md를 읽고 크로스 프로젝트 현황을 구조화.

트리거: "프로젝트 현황", "크로스 프로젝트", "전체 상태"

절차:
1. 지정된 프로젝트 디렉토리들의 STATE.md 읽기
2. 각 프로젝트별 {프로젝트명, 현재상태, 진행중, 다음할것} 추출
3. JSON 출력 + _meta 블록

출력 스키마:
```json
{
  "projects": [
    {
      "name": "orchestration",
      "status": "v3.2 안정",
      "in_progress": ["항목"],
      "next": ["항목"]
    }
  ],
  "_meta": {}
}
```

##### (4) news-verifier

위치: `~/.gemini/skills/news-verifier/SKILL.md`

목적: tech-review 포스트의 뉴스를 Google 검색으로 팩트체크.

트리거: "뉴스 검증", "팩트체크", "tech-review QA"

절차:
1. 입력된 마크다운에서 뉴스 항목 파싱 (## 섹션별)
2. 각 뉴스 항목의 핵심 주장을 Google 검색으로 교차 확인
3. freshness 체크: 실제로 최근 뉴스인지
4. coverage gap 체크: 해당 분야에서 더 큰 뉴스를 놓쳤는지
5. JSON 출력 + _meta 블록

출력 스키마:
```json
{
  "items": [
    {
      "headline": "원본 제목",
      "verified": true|false,
      "freshness": "24h이내|1주이내|오래됨",
      "notes": "검증 결과 메모",
      "alternative_source": "발견된 대안 소스 URL"
    }
  ],
  "coverage_gaps": ["놓친 주요 뉴스"],
  "_meta": {}
}
```

### 3.2 Claude Code 측 에이전트 재설계

#### 3.2.1 gemini-analyzer → 벌크 추출기

역할 변경: "전체 코드베이스 분석" → "대량 파일 구조화 추출 + 웹 검색"

호출 방식:
```bash
# 시스템 스캔 (Gemini skill 자동 활성화)
gemini -m gemini-3.1-pro-preview -p "system-scanner로 시스템 스캔해" --output-format json --yolo 2>/dev/null

# 프로젝트 스캔
gemini -m gemini-3.1-pro-preview -p "project-scanner로 /c/dev/01_projects/02_portfolio 스캔해" --output-format json --yolo 2>/dev/null

# 뉴스 팩트체크
gemini -m gemini-3.1-pro-preview -p "news-verifier로 이 콘텐츠 검증해: [콘텐츠]" --output-format json --yolo 2>/dev/null
```

#### 3.2.2 codex-reviewer → 정밀 검증기

역할 변경: "8관점 설계 결함" → "diff 리뷰 + 포맷 QA + git 분석"

호출 방식:
```bash
# diff 리뷰 (코드 리뷰 체인)
cd [프로젝트] && codex review --uncommitted \
  --dangerously-bypass-approvals-and-sandbox --ephemeral 2>&1

# git 히스토리 추출 (컨텍스트 오프로딩)
codex exec -p extract --ephemeral \
  --dangerously-bypass-approvals-and-sandbox \
  "최근 7일 git log를 JSON으로 구조화: {commits: [{date, message, files_changed}]}" 2>&1

# tech-review 콘텐츠 QA
codex exec -p verify --ephemeral \
  --dangerously-bypass-approvals-and-sandbox \
  "이 마크다운의 Smart Brevity 준수, KO/EN 일치, 말투 일관성 체크: [콘텐츠]" 2>&1
```

#### 3.2.3 ai-synthesizer → adversarial verifier

역할 변경: "교차 검증 합성" → "completeness check + 반박 검증 + verify barrier"

새로운 검증 흐름:
```
외부 CLI 출력 수신
  │
  ├─ 1단계: _meta 정합성 체크
  │   files_scanned 수치가 실제 파일 수와 일치하는가?
  │   skipped 항목이 있는가? → 해당 파일 직접 읽기
  │
  ├─ 2단계: 스팟체크 (2~3개 항목)
  │   추출된 JSON의 임의 항목을 원본 파일과 대조
  │
  └─ 3단계: 반박 검증 (adversarial)
      "이 결과에서 빠진 것은 없는가?"
      "두 CLI가 모두 놓친 blind spot은 없는가?"
```

### 3.3 트랙별 상세 흐름

#### 트랙 1: 컨텍스트 오프로딩

**세션 시작 시 시스템 파악 흐름:**

```
사용자: "시스템 파악해" 또는 /dispatch

Claude Code:
  ├─ Gemini system-scanner 호출 (백그라운드)
  │   → 에이전트/스킬/훅 메타데이터 JSON
  │
  ├─ Gemini state-scanner 호출 (백그라운드)
  │   → 4개 프로젝트 STATE 요약 JSON
  │
  ├─ Codex git 히스토리 호출 (백그라운드)
  │   → 최근 7일 변경사항 JSON
  │
  └─ Claude Opus (수신 + 검증):
      1. _meta 정합성 체크
      2. 스팟체크 1~2개
      3. 시스템 파악 완료 (총 ~25K 토큰 사용, 직접 읽기 대비 88% 절약)
```

**프로젝트 작업 진입 시:**

```
사용자: "portfolio 작업하자"

Claude Code:
  ├─ Gemini project-scanner 호출
  │   → 파일트리 + exports + types + routes JSON
  │
  └─ Claude Opus (수신):
      1. 구조 파악
      2. 수정 대상 파일 2~3개만 직접 Read
      3. 구현 시작
```

#### 트랙 2: tech-review QA

```
GitHub Actions → 포스트 생성 + 배포 (기존 유지)
  │
  ▼
Claude Code (수동: /tr-verify 또는 /morning 연동):
  │
  ├─ git pull (최신 포스트)
  │
  ├─ Gemini news-verifier (병렬)
  │   → 팩트체크 JSON (freshness, coverage gap)
  │
  ├─ Codex content-qa (병렬)
  │   → 포맷 QA JSON (Smart Brevity, KO/EN, 말투)
  │
  └─ Claude Opus (최종):
      1. 두 결과 합산
      2. 수정 필요 사항 심각도별 분류
      3. 자동 수정 가능한 것은 바로 수정
      4. 사용자에게 보고
```

#### 트랙 3: 병렬 코드 리뷰

```
implement 완료
  │
  ├─ code-reviewer(Opus) — 메인 리뷰 (항상 실행)
  │
  ├─ codex review --uncommitted (선택적, 백그라운드)
  │   조건: diff 50줄+ 또는 보안 관련 또는 /cross-review 호출
  │
  └─ Claude Opus (통합 판단):
      1. Opus 리뷰 결과 = 메인
      2. Codex 결과에서 새 발견 있으면 → Claude 검증 후 반영
      3. Codex 결과가 오탐이면 → 무시
      4. commit-writer 호출
```

### 3.4 자원 관리 규칙

#### Codex (Plus $20, 5시간 롤링)

- 한 세션에 **최대 3~5회 호출** (아껴쓰기)
- 대량 읽기 시키지 않음 (Gemini 역할)
- 프로필별 reasoning 조절로 토큰 효율화
  - extract (high): 정확도 필요 시
  - verify (low): 빠른 확인 시
  - review (medium): diff 리뷰 시
- 한도 소진 시: Claude가 직접 수행 또는 Gemini 대체

#### Gemini (AI Pro $20, 일일 rate limit)

- 대량 읽기/추출의 1순위
- 웹 검색의 1순위 (Google 엣지)
- skills로 절차 고정 → 자유도 최소화
- --output-format json 강제
- --yolo 모드 (자동 승인, Claude Code 내에서 실행이므로)
- 해석/판단 절대 시키지 않음

#### Claude Opus (1M 컨텍스트, 중심)

- 설계/결정/코드 작성 = Claude 전권
- 외부 CLI 결과의 최종 판단자
- 컨텍스트는 실제 작업(설계, 구현, 대화)에 집중
- 읽기 그런트워크는 외부 CLI에 위임

### 3.5 Verify Barrier 설계

모든 외부 CLI 출력에 적용되는 3단계 검증:

```
[1단계: 구조 검증] — 자동, 비용 ~0
  - JSON 파싱 성공 여부
  - _meta 블록 존재 여부
  - files_scanned 수치와 실제 파일 수 비교
  - skipped 항목 확인

[2단계: 스팟체크] — 선택적, 비용 ~1K 토큰
  - 추출 결과에서 임의 2~3개 항목 선택
  - 해당 원본 파일을 Claude가 직접 읽어 대조
  - 불일치 발견 시: 전체 재추출 또는 해당 부분만 직접 읽기

[3단계: 반박 검증] — 고위험 작업만, 비용 ~3K 토큰
  - 배포 전 보안 리뷰
  - 시스템 구조 변경 시
  - "이 결과에서 빠진 것이 있는가?" 자문
```

검증 생략 조건:
- 이전 세션에서 같은 스킬로 추출한 결과와 구조가 동일할 때
- _meta.completeness가 "COMPLETE"이고 skipped가 0일 때 → 1단계만

1. _meta.skipped 확인 — 빠진 파일이 있는지
2. 스팟체크 범위 확대 (2~3개 → 5개)
3. 해당 스킬의 절차를 더 세분화 (자유도 낮추기)
4. 지속적이면: 해당 작업을 Codex로 전환

**Codex 5시간 한도 도달 시:**
1. 남은 작업을 Gemini로 전환 가능한지 판단
2. 불가능하면 Claude가 직접 수행
3. 다음 세션에서 Codex 호출 횟수 조정

**CLI 미설치/오류 시:**
1. 에이전트가 `which codex`/`which gemini` 사전 확인
2. 미설치 시 설치 안내만 출력, 작업은 Claude가 직접 수행
3. API 오류 시 재시도 1회 → 실패 시 Claude 직접 수행

**a) Smart Brevity 형식**
- "Today in One Line" 섹션 존재 여부
- 각 뉴스 항목에 "Why it matters" 존재 여부
- ## 번호 섹션이 4개 이상인지
- 각 섹션이 150~400자 범위인지

**b) KO/EN 일치** (두 파일이 제공된 경우)
- 뉴스 항목 수 일치 여부
- 각 항목의 핵심 사실(기업명, 수치, 날짜)이 동일한지
- 누락된 섹션이 없는지

**c) 말투 일관성**
- "~다" 체와 "~습니다" 체가 혼용되는지
- 한 문서 내에서 통일된 어미를 사용하는지

**d) 키워드 중복**
- keywords-log.md가 제공된 경우, 이전 같은 요일과 주제가 겹치는지
- 같은 기업 + 같은 사건인지 판단

**e) 분량**
- 전체 글자 수가 2,500자 이상인지
- 각 섹션이 최소 100자 이상인지

# Cross Review — 외부 모델 병렬 코드 리뷰 스킬

Codex CLI의 review 명령으로 현재 uncommitted diff를 리뷰하고, Claude Opus 리뷰와 교차한다.

## 사용법

- `/cross-review` — 현재 디렉토리의 uncommitted 변경 리뷰
- `/cross-review [프로젝트 경로]` — 특정 프로젝트 리뷰
- `/cross-review --security` — 보안 관점 강화 리뷰

## 전제 조건

- uncommitted 변경사항이 있어야 함 (없으면 종료)
- Codex CLI 설치 필수 (미설치 시 안내 후 종료)

## 절차

### 1. 사전 확인

```bash
which codex 2>/dev/null || echo "CODEX_MISSING"
cd [프로젝트 경로] && git diff --stat
```

- Codex 미설치: "Codex CLI가 필요합니다. npm i -g @openai/codex" 출력 후 종료
- 변경사항 없음: "uncommitted 변경사항 없음" 출력 후 종료

### 2. diff 규모 확인

```bash
git diff --stat | tail -1  # N files changed, N insertions, N deletions
```

- 10줄 미만: "변경 규모가 작아 cross-review 불필요" 안내 (사용자가 원하면 계속)
- 50줄 이상: 자동 추천 메시지

### 3. 병렬 실행

#### Claude Opus 리뷰 (메인)

기존 code-reviewer 에이전트 호출 (항상 실행).

#### Codex 리뷰 (보조, 백그라운드)

```bash
cd [프로젝트 경로] && codex review --uncommitted \
  --dangerously-bypass-approvals-and-sandbox \
  --ephemeral 2>&1
```

--security 플래그 사용 시:
```bash
cd [프로젝트 경로] && codex review --uncommitted \
  --dangerously-bypass-approvals-and-sandbox \
  --ephemeral \
  "보안 관점에 집중: XSS, 인젝션, env 노출, CORS, 인증 우회, 하드코딩 시크릿" 2>&1
```

### 4. 결과 교차

Claude가 두 결과를 받아서:

1. **일치하는 이슈**: 신뢰도 높음 → 즉시 반영
2. **Codex만 발견한 이슈**: Claude가 해당 코드를 직접 읽어 검증
   - 진짜 이슈 → 반영
   - 오탐 → "Codex 오탐: [이유]"로 기록 후 무시
3. **Claude만 발견한 이슈**: Claude 판단 유지
4. **둘 다 미발견**: blind spot 인지 (추가 조치 없음)

### 5. 보고

```
=== Cross Review: [프로젝트명] ===

변경: N files, +N/-N lines

Claude Opus 발견: N건
Codex (GPT-5.3) 발견: N건

교차 결과:
  🔴 일치 (높은 신뢰): N건
    1. [HIGH] file.tsx:42 — null 참조 가능성
  🟡 Codex만 발견 → Claude 검증 완료: N건
    2. [MEDIUM] api.ts:15 — 미인코딩 쿼리 파라미터 (진짜 이슈)
    3. Codex 오탐: utils.ts:8 — 실제로는 안전 (타입 가드 존재)
  🔵 Claude만 발견: N건
    4. [LOW] style.css:22 — 미사용 클래스

총 이슈: N건 (CRITICAL: 0, HIGH: 1, MEDIUM: 1, LOW: 1)

code-reviewer 판정: [🔴/✅] → commit-writer 호출 여부
```

## 부록 C: Gemini 모델 지정 및 호출 규칙

### C.1 모델 지정

모든 Gemini CLI 호출에 반드시 `-m gemini-3.1-pro-preview` 플래그를 포함한다.

이유:
- Gemini CLI 기본 모델이 변경될 수 있음
- 일관된 추출 품질을 보장하기 위해 모델을 고정
- 새 모델로 전환 시 GEMINI.md와 이 문서를 함께 업데이트

### C.2 표준 호출 패턴

```bash
# 기본 패턴 (모든 추출 작업)
gemini -m gemini-3.1-pro-preview \
  -p "[프롬프트]" \
  --output-format json \
  --yolo \
  2>/dev/null

# 웹 검색 포함 (news-verifier 등)
gemini -m gemini-3.1-pro-preview \
  -p "[프롬프트]" \
  --output-format json \
  --yolo \
  2>/dev/null

# 특정 디렉토리 포함 (프로젝트 스캔)
gemini -m gemini-3.1-pro-preview \
  -p "[프롬프트]" \
  --output-format json \
  --yolo \
  --include-directories [추가 디렉토리] \
  2>/dev/null
```

### C.3 플래그 설명

| 플래그 | 값 | 이유 |
|--------|-----|------|
| `-m` | `gemini-3.1-pro-preview` | 모델 고정 (변경 방지) |
| `-p` | 프롬프트 문자열 | 비대화형(headless) 모드 |
| `--output-format` | `json` | 구조화 출력 강제 |
| `--yolo` | (플래그) | 자동 승인 (Claude Code 내부이므로 안전) |
| `--include-directories` | 경로 | 워크스페이스 외 디렉토리 포함 |
| `2>/dev/null` | — | stderr 억제 (로딩 메시지 등) |

### C.4 Codex 표준 호출 패턴

```bash
# 구조화 추출 (extract 프로필)
codex exec -p extract --ephemeral \
  --dangerously-bypass-approvals-and-sandbox \
  -C [작업 디렉토리] \
  "[프롬프트]" 2>&1

# 빠른 검증 (verify 프로필)
codex exec -p verify --ephemeral \
  --dangerously-bypass-approvals-and-sandbox \
  "[프롬프트]" 2>&1

# diff 리뷰 (review 명령)
cd [프로젝트] && codex review --uncommitted \
  --dangerously-bypass-approvals-and-sandbox \
  --ephemeral 2>&1

# 프롬프트 템플릿 사용
codex exec -p extract --ephemeral \
  --dangerously-bypass-approvals-and-sandbox \
  "/prompts:[템플릿명] ARG1=val1 ARG2=val2" 2>&1
```

### C.5 플래그 설명

| 플래그 | 값 | 이유 |
