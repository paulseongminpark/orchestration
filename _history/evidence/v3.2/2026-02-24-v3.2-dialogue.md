## Round 6: 리좀 연결자 심화 + 크로스팀 강화

### 리좀 연결자 — 팀을 만들어야 하나, 소속 없음이 나은가?

**사용자**: 리좀 연결자도 팀을 따로 만들거나 리드를 만드는 게 나은가? 소속 없음으로 가야 하는가? 최적의 운영은?

**Claude**: **소속 없음(floating)이 최적이다.** 이유:

1. **시냅스의 본질**: 시냅스는 특정 뉴런(팀)에 소속되지 않는다. 뉴런 사이를 연결하는 것이 역할. 팀에 가두면 그 팀의 트리거에만 반응하게 됨 → 유연성 상실.

2. **트리거 다양성**: context-linker는 SessionStart hook에서, project-linker는 commit-writer 후에 발동. 서로 다른 팀의 이벤트에 반응해야 함. 한 팀에 넣으면 다른 팀의 이벤트에 접근이 복잡해짐.

3. **경량성**: Haiku/Sonnet. 팀 관리 오버헤드(TeamCreate, TaskList, SendMessage) 없이 직접 Task 호출이 더 빠르고 가벼움.

4. **팀을 만들면 생기는 문제**:
   - 리드가 필요한데, 연결자 2개에 리드를 지정하면 오버헤드만 증가
   - 팀 태스크 관리 불필요 (이들은 "작업"이 아니라 "전달"을 함)
   - 팀 간 소통이 아니라 팀을 관통하는 것이 목적

**최적 운영**:
- 팀이 아닌 **"메시(mesh)"** 로 정의
- KNOWLEDGE.md에 메시 규칙 기록
- 각 연결자의 트리거는 해당 체인(hook, 에이전트)에 직접 배선
- 연결자끼리의 소통: live-context.md를 통한 간접 소통 (직접 소통 불필요)

### 리좀 연결자 + 크로스팀 유틸리티 — 놓치는 부분?

**사용자**: 면밀히 검토했을 때 놓치는 부분은? 더 만들거나 상세하게?

**Claude**: 현재 커버리지 점검:

| 영역 | 현재 담당 | 놓치는 부분 |
|------|----------|-----------|
| 세션↔세션 맥락 | context-linker | ✅ 커버 |
| 프로젝트↔프로젝트 영향 | project-linker | ✅ 커버 |
| 커밋 | commit-writer | ✅ 커버 |
| 방향/상태 | orch-state | ⚠️ decisions.md 현재 프로젝트 관련 항목 체크 추가 필요 |
| 컨텍스트 파악 | project-context | ✅ 커버 |
| 글쓰기 | content-writer | ✅ 커버 |
| **결정 흐름** | **없음** | ❌ decisions.md 미반영 항목이 해당 프로젝트에 자동 알림되지 않음 |
| **학습 전파** | **없음** | ❌ analyze 팀 결과가 agent.md에 자동 반영되지 않음 |
| **에러 복구** | **없음** | ❌ 에이전트 실패 시 복구 메커니즘 없음 |
| **건강 모니터링** | **부분** | ⚠️ session-start가 기본 체크만. hook 정상 작동 확인 없음 |

**강화 제안 (기존 에이전트에 역할 추가)**:

1. **orch-state에 "결정 워처" 역할 추가**:
   - 현재: STATE.md 분석 + 방향 제안
   - 추가: decisions.md에서 현재 프로젝트 관련 미반영 항목 스캔
   - 출력: "⚠️ [portfolio] 미반영 결정 2건: Obsidian 반응형 미확인, 스크린샷 이미지 링크"
   - 이렇게 하면 orch-state가 "상태+결정" 통합 분석기가 됨
   - 토큰 추가: +100 (decisions.md 스캔)

2. **ai-synthesizer에 "학습 전파자" 역할 추가**:
   - 현재: 교차 검증 결과 합의/불일치 판단
   - 추가: 합의된 학습 항목 → 해당 agent.md의 "학습된 패턴" 섹션에 직접 제안
   - 출력: `code-reviewer.md에 추가 제안: "Tailwind CSS 4에서 @apply 미지원, 인라인 클래스 우선"`
   - 사용자 승인 후 반영 (자동 쓰기 아닌 제안만)

3. **doc-syncer에 "건강 모니터" 역할 추가**:
   - 현재: STATE.md vs 실제 파일 검증 (3레이어)
   - 추가: hook 파일 존재 확인 + settings.json에 등록 확인 + 스크립트 실행 권한 확인
   - 출력: "✅ 전체 hook 정상" 또는 "❌ post-tool-live-context.sh 실행 권한 없음"

**시냅스를 더 빠르고 동적이고 반응성 좋게 만드는 방법**:

| 방법 | 적용 | 효과 |
|------|------|------|
| **모든 연결자를 Haiku로** | context-linker: 이미 Haiku. project-linker: Sonnet→Haiku 고려 | 응답 속도 2~3배 향상 |
| **트리거 다양화** | project-linker: commit 외에도 "git add" 시점에 pre-scan | 더 이른 감지 |
| **선택적 실행** | project-linker: 파일 경로에 다른 프로젝트 키워드 없으면 skip | 불필요 실행 제거 |
| **캐시** | context-linker: 이전 스캔 결과 live-context.md 헤더에 캐시 | 반복 스캔 방지 |
| **점진적 스캔** | context-linker: 전체 스캔 대신 마지막 스캔 이후 추가분만 | 토큰 절감 |

**project-linker Haiku 전환 검토**:
- 장점: 3초→1초. 커밋마다 발동되므로 빈도 높음 → 절감 효과 큼
- 단점: 연관 맵 기반 키워드 매칭은 Haiku도 충분. Sonnet의 "구조적 이해"는 키워드 기반에선 불필요.
- **추천: Haiku로 전환**. 연관 맵은 명시적 규칙이므로 Haiku가 정확히 처리 가능.
