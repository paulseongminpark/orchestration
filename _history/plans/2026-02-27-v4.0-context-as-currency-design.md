## 1. 문제 의식과 비전

### 문제

v3.3은 **1M context window** 전제로 설계됐다. 하지만 1M 사용 시 항상 extra usage가 발생하므로 **200K context**에서 운영 중이다. 그 결과:

- Baseline ~42K (에이전트 24개 + 스킬 14개 descriptions + hooks + MEMORY.md + using-superpowers)
- 세션당 실질 작업 가능 토큰 부족 → 잦은 compact
- 에이전트/스킬이 per-turn 주입되어 매 턴 컨텍스트 소비

### 비전

**"Context as Currency"** — 모든 토큰은 비용이다. 필요할 때만 로드하고, 불필요하면 격리한다.

- **Claude Code** = 메인 오케스트레이터 (의사결정 + verify + merge)
- **Gemini CLI / Codex CLI** = 독립 Warp pane에서 worktree 기반 병렬 작업
- **Git 브랜치**로 결과 수렴, Claude Code가 verify barrier로 검증
- **rulesync**로 모든 CLI의 규칙 파일을 단일 소스에서 자동 생성
- **Google Context Engineering** 패턴 적용: Attention Manipulation, Preserve Failures, Consolidation

### v3.3.1 → v4.0 변경 요약

| 항목 | v3.3.1 | v4.0 |
|------|--------|------|
| 에이전트 | 24개 | **15개** (4삭제 + 5통합) |
| 스킬 | 14개 (per-turn 주입) | **9개** (disable-model-invocation) |
| 팀 | 4팀+허브 | **3팀+허브** |
| CLI 통합 | Gemini/Codex 에이전트 래핑만 | **worktree + branch + verify** |
| 규칙 관리 | 수동 (CLAUDE.md/GEMINI.md 각각) | **rulesync 자동 생성** |
| 메모리 | 3-phase (추출→이동→정리) | **4-phase (+Consolidation)** |
| Baseline | ~42K | **~28K** (33% 절감) |

```yaml
## v4.0 강화 규칙 (Google Context Engineering 적용)
1. Attention Manipulation: compact 요약 마지막에 "세션 목표 + 남은 할 일" 반드시 재작성
   → compact 후 자동 Read 시 목표가 가장 최근 context에 위치
   → "lost-in-the-middle" 완화
2. Preserve Failures: 실패 기록 삭제 금지
   → "시도→실패→원인" 3줄은 반드시 보존
   → 같은 실수 반복 방지
3. Restorable Compression: URL/파일경로만 보존, 내용은 drop
   → 필요 시 원본 복구 가능
```

## 4. Gemini CLI 구성

### 4.1 Gemini의 역할

Gemini CLI는 Claude Code의 "팀원"이 아니다. **독립 작업자**다.
Claude Code 에코시스템(에이전트/팀/훅)에는 Gemini 개념이 없다.
Git 브랜치를 통해서만 연결된다.

**Gemini가 할 수 있는 것:**
- 독립 worktree에서 자유 작업 (기능 한정 없음)
- Conductor로 구조적 개발 (spec → plan → implement → review)
- Skills로 전문 지식 온디맨드 로드
- 내장 Specialist 자동 활성화 (codebase_investigator, browser_agent)
- 1M+ 컨텍스트로 대규모 분석

**Gemini가 할 수 없는 것:**
- Claude Code의 에이전트/팀/훅 참여
- Claude Code TaskList 접근
- Claude Code memory:user 사용

### 4.2 디렉토리 구조

```
글로벌:
~/.gemini/
├── settings.json               # 전역 설정
└── skills/                     # 글로벌 스킬

프로젝트별:
/c/dev/
├── GEMINI.md                   # 전역 지침 (rulesync 생성)
└── 01_projects/{project}/
    ├── GEMINI.md               # 프로젝트 지침 (rulesync 생성)
    └── .gemini/
        ├── settings.json       # 프로젝트 오버라이드
        └── skills/             # 프로젝트 스킬
            ├── system-scanner/SKILL.md
            ├── project-scanner/SKILL.md
            ├── state-scanner/SKILL.md
            ├── news-verifier/SKILL.md
            └── bulk-extract/SKILL.md    # v4.0 신규
```

### 4.3 settings.json 상세

```json
{
```

### 4.4 Gemini Skills (Progressive Disclosure)

Gemini Skills의 3단계 수명주기:
1. **Discovery**: 세션 시작 → 스킬 이름+설명만 시스템 프롬프트에 주입 (최소 토큰)
2. **Activation**: 에이전트가 필요 판단 → `activate_skill` 호출 → 사용자 승인
3. **Execution**: SKILL.md 전체 + scripts/ + references/ 로드

이것이 Claude Code의 `disable-model-invocation: true`와 **동일한 효과**. Gemini는 네이티브 지원.

### 4.5 Conductor (구조적 개발)

Gemini가 worktree에서 독립 작업할 때 사용하는 워크플로우 관리 확장:

```
conductor/
├── context.md                   # 제품 목표, 기술 스택, 가이드라인
└── tracks/
    └── {task-id}/
        ├── spec.md              # 요구사항 + 제약조건
        ├── plan.md              # 단계별 태스크 + 체크리스트
        └── review.md            # 자동 생성: 코드 품질 검증 리포트
```

워크플로우:
```
/conductor:newTrack "{task}" → spec.md 생성 → 사용자 승인
→ plan.md 생성 → 사용자 승인
→ /conductor:implement → 코드 작성
→ Automated Review → 정적 분석 + 스타일 검증
→ branch commit+push
→ Claude Code verify barrier
```


## 5. Codex CLI 구성

### 5.1 Codex의 역할

Codex CLI도 **독립 작업자**. Claude Code 에코시스템 밖에서 동작한다.
다만 Codex는 Gemini보다 설정이 더 풍부하다: **AGENTS.md 체인 + Skills + MCP + 프로필**.

### 5.2 AGENTS.md 3단 체인 (검증된 공식 기능)

Codex CLI는 시작 시 AGENTS.md를 **계층적으로 체인 로드**한다:

```
Layer 1: ~/.codex/AGENTS.override.md     # 전역 불변 규칙 (항상 적용)
    ↓ concat
Layer 2: /c/dev/AGENTS.md               # 프로젝트 규칙 (rulesync 생성)
    ↓ concat
Layer 3: <worktree>/AGENTS.override.md   # worktree별 태스크 규칙 (작업 시 자동 생성)
    ↓
    = Codex가 읽는 최종 instructions (합산 최대 32KiB)
```

**Layer 1 — 전역 불변** (`~/.codex/AGENTS.override.md`):
```markdown
# GLOBAL Codex Rules (불변)
- 한국어 출력
- Git push 금지. main/master 직접 커밋 금지.
- 작업 결과는 branch에 commit만.
- 출력에 항상 _meta 블록 포함 (검증용)
- 위험 명령 금지: rm -rf, force push, .env 수정
- 작업 전: .ctx/shared-context.md 읽기
```

**Layer 2 — 프로젝트** (`/c/dev/AGENTS.md`, rulesync 생성):
```markdown
# Project Rules (rulesync 생성)
## SoT: STATE.md (인벤토리), CLAUDE.md (규칙)
## 프로젝트: orchestration(main), portfolio(master)
## 스택: Next.js 15, Tailwind 4, shadcn/ui, TypeScript
## 커밋: [project] 한줄 설명 + Co-Authored-By
## 테스트: npm run build && npm run lint
```

**Layer 3 — Worktree** (`<worktree>/AGENTS.override.md`, 작업 시 생성):
```markdown
# Worktree Override
TASK: auth-refactor
ROLE: codex-implement
BRANCH: codex/auth-refactor
SCOPE: src/lib/auth/, src/app/api/auth/
NON_GOALS: UI 변경, 스타일 수정, 문서
OUTPUT: 완료 시 branch commit.
```

### 5.3 Codex 프로필 (config.toml)

```toml
# ~/.codex/config.toml

[profile.review]
model = "o4-mini"
approval_mode = "suggest"          # 변경 제안만

[profile.implement]
model = "o4-mini"

[profile.extract]
model = "o4-mini"

[profile.verify]

[mcp_servers.context7]
```

**사용법:**
```bash

codex exec -p verify       # 포맷 QA
```

**제한**: 5시간 롤링, 세션당 3~5회 권장.

## 7. rulesync — 통합 규칙 관리

### 7.1 왜 필요한가

5개 pane × 3종 CLI = 최소 3개 규칙 파일 (CLAUDE.md, GEMINI.md, AGENTS.md).
규칙 하나 바꿀 때 3개 파일 수동 편집? → rulesync가 해결.

### 7.2 검증된 사실

| 항목 | 값 |
|------|-----|
| 리포 | [dyoshikawa/rulesync](https://github.com/dyoshikawa/rulesync) |
| 버전 | v7.9.0 (2026-02-27) |
| Stars | 831, npm ~161K 다운로드/주 |
| 지원 도구 | **21개** (Claude Code, Gemini CLI, Codex CLI, Cursor, Copilot, Warp 등) |
| 기능 | rules, commands, MCP, skills, hooks, subagents |

### 7.3 사용법

```bash
npm install -g rulesync

# 초기화
rulesync init                    # .rulesync/ + rulesync.jsonc 생성

# 기존 설정 가져오기
rulesync import --targets claudecode

# 전체 도구에 생성
rulesync generate --targets "*" --features "*"

# 특정 도구만
rulesync generate --targets claudecode,geminicli,codexcli --features rules
```

### 7.4 우리 시스템에서의 구조

```
/c/dev/
├── .rulesync/                    # SoT (사람이 편집)
│   ├── rules/
│   │   ├── global.md             # 공유 규칙 (언어, Git, 보안)
│   │   ├── orchestration.md      # orch 규칙
│   │   └── portfolio.md          # portfolio 규칙
│   └── rulesync.jsonc            # 대상 도구 설정
├── CLAUDE.md                     # ← rulesync 생성 (공유 부분) + 수동 (체인/에이전트)
├── GEMINI.md                     # ← rulesync 생성
└── AGENTS.md                     # ← rulesync 생성
```

**주의**: CLAUDE.md에는 rulesync가 관리하는 "공유 규칙" + Claude 전용 "체인/에이전트" 공존.
rulesync는 자기 영역(마커로 구분)만 건드리고, 수동 섹션은 보존.

### 7.5 규칙 변경 워크플로우

```
1. .rulesync/rules/global.md 수정 (예: 새 Git 규칙 추가)
2. rulesync generate --targets "*"
3. CLAUDE.md, GEMINI.md, AGENTS.md 자동 갱신
4. git commit + push
5. 모든 pane이 다음 세션부터 새 규칙 적용
```

## 8. 메모리 & 세션 관리

### 8.1 메모리 유형 매핑 (Google Context Engineering 기반)

| 유형 | 우리 구현 | 파일 | 범위 |
|------|-----------|------|------|
| **Short-term** | 세션 context window | (메모리 내) | 현재 세션 |
| **Long-term** | Auto Memory | MEMORY.md (200줄 캡) | 세션 간 |
| **Episodic** | 세션 기록 | session-summary.md, .chain-temp/ | 이력 |
| **Semantic** | 지식 | KNOWLEDGE.md | 영구 |
| **Procedural** | 프로세스 | CLAUDE.md (체인, 규칙) | 영구 |
| **Cross-agent** | 에이전트 간 | live-context.md (100줄 캡) | 세션 간 |
| **Cross-CLI** (v4.0) | CLI 간 | **.ctx/shared-context.md** | 실시간 |

### 8.2 메모리 파이프라인 (3-phase → 4-phase)

```
Phase 1: Extraction (추출)
  트리거: SessionEnd hook → analyze-session.sh
  결과: pending.md에 추출된 메모리 항목

Phase 2: Transfer (이동)
  트리거: /sync-all → sync-memory.sh
  결과: pending.md → MEMORY.md로 이동

Phase 3: Consolidation (통합) ← v4.0 신규
  트리거: /memory-review (주간) 또는 자동
  수행:
    - 중복 감지: 기존 MEMORY.md 항목과 의미적 비교
    - 모순 해결: 새 정보가 기존과 충돌 시 최신 우선
    - TTL 망각: 2주 이상 참조 안 된 항목 → 제거 후보 표시
    - 진화 추적: 단순 사실 → 뉘앙스 이해로 업데이트
  도구: LLM 기반 (memory-review 스킬 강화)

Phase 4: Maintenance (정리)
  트리거: /memory-review (수동)
  수행: 사용자 승인 후 구식/중복 항목 제거
```

### 8.4 Compaction 전략 (Google 백서 적용)

| 전략 | 설명 | 우리 구현 |
|------|------|-----------|
| **Restorable Compression** | URL/경로만 보존, 내용 drop → 필요 시 복구 | .chain-temp 패턴 |
| **Attention Manipulation** | compact 시 목표+todo 재작성 → 최근 attention에 밀어넣기 | compressor v4.0 강화 |
| **Preserve Failures** | 실패 기록 삭제 금지 → 같은 실수 방지 | compressor v4.0 강화 |
| **Token Truncation** | 100K/120K/150K 3단계 임계값 | 유지 |
| **Recursive Summarization** | compressor 9단계 | 유지 |

### 8.5 Memory Trigger 설계

| 트리거 | 시점 | 수행 |
|--------|------|------|
| **Session completion** | SessionEnd hook | analyze-session.sh → pending.md |
| **Explicit command** | /compact, /sync | compressor, sync 스킬 |
| **Task completion** (v4.0) | TaskCompleted hook | .ctx/shared-context.md 갱신 + pending.md에 학습 항목 |
| **Cross-CLI event** | Gemini/Codex branch push | linker가 감지 → shared-context.md 갱신 |

## 10. Context Engineering 12대 규칙

```
 1. 읽지 않은 파일은 존재하지 않는다      — 추측 금지, Read 후 판단
 2. 에이전트는 결과만 반환한다             — 과정은 .chain-temp, 메인에 1줄
 3. 스킬은 호출될 때만 존재한다            — disable-model-invocation으로 평소 비용 0
 4. 외부 출력은 유죄 추정                  — Verify Barrier 3단계 통과 전 사용 금지
 5. 1세션 = 1목표                          — 목표 달성 후 세션 전환 체인
 6. 100K = 위험, 120K = 필수 compact       — 150K auto-compact은 최후 방어선
 7. 탐색은 서브에이전트                     — 메인 context는 의사결정 전용
 8. 경량은 직접 호출, 중량은 worktree       — CLI 호출 비용 최적화
 9. 로컬 .md가 SoT, Git은 동기화           — STATE.md=인벤토리, CLAUDE.md=규칙
10. rulesync가 규칙의 SoT                  — CLI별 파일은 자동 생성물
11. compact 시 목표 + 실패를 재작성         — Attention Manipulation + Preserve Failures
12. 기억에는 출처를 표시한다                — [claude], [gemini], [codex] 마커
```
