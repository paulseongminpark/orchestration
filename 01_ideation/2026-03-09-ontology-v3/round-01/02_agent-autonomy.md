# 02: 에이전트 자율성 → 시스템 반영

> Paul: "에이전트의 본질은 목표 지향적 자율성이다. 지시를 받아 실행하는 게 아니라
> 목표를 이해하고, 방법을 스스로 선택하고, 결과를 평가하고, 방향을 수정한다."

---

## 1. 현재 시스템의 자율성 수준

### 1-1. 현재 구조 (규칙 지향)

```
CLAUDE.md → 규칙 읽기
rules/*.md → IF 조건 THEN 행동
MEMORY.md → 과거 맥락 참조
hooks → 이벤트 트리거 → 규칙 실행
```

Claude는 **규칙을 따르는** 실행자:
- "커밋 후 push" = 규칙
- "compact 전 3곳 저장" = 규칙
- "recall 먼저" = 규칙

**문제**: 규칙이 없으면 행동하지 않는다. 규칙이 100개가 되면 관리 불가.

### 1-2. 목표 지향 구조 (필요한 것)

```
목표 이해: "Paul이 지금 뭘 하려는가"
맥락 파악: "이전에 비슷한 걸 해봤는가"
방법 선택: "지금 상황에서 최선은 뭔가"
결과 평가: "내가 한 게 목표에 맞았는가"
방향 수정: "다음엔 어떻게 할까"
```

### 1-3. 현실적 제약

- Claude는 세션마다 리셋 (stateless)
- 자율 판단에는 충분한 맥락이 필요
- 맥락 로드 = 토큰 비용
- 잘못된 자율 판단 = 사고 (3-05 멀티세션 사고)

---

## 2. 자율성 레벨 정의

| Level | 이름 | 현재 | 설명 |
|-------|------|------|------|
| L0 | 명시 실행 | ✅ | "이거 해" → 한다 |
| L1 | 규칙 기반 | ✅ | 규칙 읽고 따른다 |
| L2 | 맥락 참조 | 부분 | recall()로 과거 참조 후 판단 |
| L3 | 목표 추론 | ❌ | 사용자 의도를 추론하고 방법 선택 |
| L4 | 자기 평가 | ❌ | 결과를 평가하고 다음 행동 조정 |
| L5 | 선제적 행동 | ❌ | 필요한 걸 사용자가 말하기 전에 함 |

현재: L1~L2. 목표: L3까지.

L4-L5는 위험 (3-05 사고 = L5 시도의 실패). 당분간 L3이 적정선.

---

## 3. L3 "목표 추론" 구현 방향

### 3-1. 목표의 계층

```
L0: 즉시 목표 — "이 파일 수정해"
L1: 세션 목표 — "NDCG 개선" (dispatch/TASK_CONTRACT)
L2: 프로젝트 목표 — "mcp-memory를 active recall 시스템으로" (STATE.md)
L3: 근본 목표 — "뇌의 다차원 연결을 외부화" (Identity/Narrative)
```

현재 시스템이 아는 것:
- L0: 사용자 메시지에서
- L1: TASK_CONTRACT, dispatch에서 (있으면)
- L2: STATE.md에서 (있으면)
- L3: mcp-memory Identity/Narrative에서 (recall 필요)

**문제**: L1-L3를 자동으로 로드하는 구조가 없다.

### 3-2. 목표 인식 파이프라인

```
세션 시작
  → get_context() — 현재 프로젝트 목표 (L2)
  → 사용자 첫 메시지 분석 — 즉시 목표 (L0)
  → recall(L0 키워드) — 관련 과거 맥락
  → L0 + L2 + recall 결과 → 세션 목표(L1) 자동 구성
  → 행동 선택: 규칙 + 목표 기반
```

vs 현재:
```
세션 시작
  → CLAUDE.md, rules 로드 (자동)
  → get_context() 200토큰 (자동)
  → 사용자 메시지 → 규칙 매칭 → 실행
```

**차이**: 목표 인식 파이프라인은 recall()을 적극 활용.
Pre-flight Recall(S1에서 구현)이 이것의 첫 단계.

### 3-3. 방법 선택의 자율성

현재: "커밋해라" → commit-writer 호출 (규칙)
목표 지향: "구현 끝났다" → (목표: 품질 보장) → code-review 필요 판단 → code-reviewer 호출 → 통과 → commit-writer

이건 이미 workflow.md 체인으로 구현되어 있다:
> "구현 완료 감지 → code-reviewer 자동"

**통찰**: 체인은 규칙이지만, 체인의 트리거를 목표 기반으로 바꾸면 L3.

### 3-4. mcp-memory의 역할

목표 지향 에이전트에서 mcp-memory는 **장기 기억**:
- 과거 실패 → 같은 실수 반복 방지
- 과거 결정 → 일관성 유지
- 과거 패턴 → 최선의 방법 선택

**핵심**: recall()의 품질 = 에이전트 자율성의 품질.
NDCG가 0.724인 건 좋지만, 그건 "비슷한 텍스트 찾기"의 품질.
목표 지향에 필요한 건 "이 상황에서 필요한 기억 찾기" = **의미적 인출**.

---

## 4. 의미적 인출 vs 텍스트 인출

| | 텍스트 인출 (현재) | 의미적 인출 (목표) |
|---|---|---|
| 쿼리 | "에이전트 워크플로우" | "구현 끝났는데 다음 뭐하지?" |
| 매칭 | 벡터 유사도 | 상황 매칭 |
| 결과 | 비슷한 단어가 포함된 노드 | 지금 상황에서 필요한 노드 |
| 예시 | Workflow 타입 노드들 | 과거 유사 상황의 Decision + Failure |

**01의 retrieval_hints가 이 격차를 줄인다:**
- `when_needed: "구현 완료 후"` → 상황 매칭 가능
- `related_queries: ["다음 단계", "리뷰"]` → 의도 매칭 가능

## 5. 구현 제안

### 단기 (L2 강화)
1. session-start에 recall(프로젝트 목표 키워드) 추가 — 이미 Pre-flight Recall로 부분 구현
2. TASK_CONTRACT 자동 생성 (사용자 첫 메시지 + recall 결과)

### 중기 (L3 진입)
3. intent 인식: 사용자 메시지에서 즉시 목표(L0) + 암묵적 목표 추론
4. recall()에 intent 전달 → intent 기반 검색 전략 선택 (01과 연결)
5. 체인 트리거를 목표 기반으로 전환 (규칙 → 목표 매칭)

### 장기 (L3 안정화)
6. 결과 평가 피드백 루프: 세션 끝에 "목표 달성도" 기록
7. 달성도 기반 recall 가중치 조정 (BCM 학습률 활용)
