# R2-02: 에이전트 자율성 L3 — 구현 경로

> R1: L0-L5 프레임워크, 현재 L1-L2, 목표 L3.
> Phase 0: patch foraging, classify→store→link 분리.
> 이 문서: L3 안전 경계 + 구체적 구현 설계.

---

## 1. L3 "목표 추론"이란

사용자가 "이거 해"라고 안 했는데, 맥락에서 필요한 행동을 추론하는 것.

현재 (L1-L2):
```
사용자: "enrichment 돌려"
Claude: enrichment 실행 (L0 명시 실행)
       → 규칙에 따라 NDCG 재측정 (L1 규칙 기반)
       → recall()로 이전 enrichment 결과 참조 (L2 맥락 참조)
```

L3:
```
사용자: "다음 뭐하지"
Claude: → dispatch로 세션 맥락 파악 (L2)
       → recall("현재 프로젝트 미결") (L2)
       → 미결 중 가장 impact 높은 것 추론 (L3 목표 추론)
       → "NDCG가 0.724인데 ZERO 10건 중 goldset 오류 2건 수정하면 바로 개선됩니다" (L3)
```

## 2. L3 안전 경계 (3-05 사고 교훈)

### 2-1. 3-05 사고 복기

범용 프롬프트("인덱스 읽고 이어서") → 세션 B가 소스코드 수정+커밋, 세션 C가 config 수정+커밋.
**원인**: L5(선제행동)를 시도한 것. 세션이 "이어서 해야 할 것"을 스스로 판단하고 실행.

### 2-2. L3 vs L5 경계

| | L3 (안전) | L5 (위험) |
|---|----------|----------|
| 행동 | **제안**한다 | **실행**한다 |
| 범위 | 현재 세션 + 현재 프로젝트 | 다른 세션/프로젝트에 영향 |
| 확인 | 사용자 승인 후 실행 | 사용자 승인 없이 실행 |
| 예시 | "goldset 수정하면 NDCG 올라갈 것 같습니다" | goldset을 직접 수정하고 커밋 |

**L3 규칙**: 추론은 자유, 실행은 제안. 코드 수정/커밋은 항상 사용자 승인.

### 2-3. L3 안전 장치

```
1. 추론 결과를 먼저 보여준다 (실행 전)
2. 영향 범위를 명시한다 ("이 파일 수정", "이 DB 변경")
3. 현재 프로젝트 밖으로 나가지 않는다
4. 파괴적 행동(delete, force-push, schema change)은 L3에서 제외
```

## 3. 구현 설계

### 3-1. 목표 인식 파이프라인

```
세션 시작
  → [자동] CLAUDE.md, rules 로드
  → [자동] get_context(200토큰) — 외부 메모리 요약
  → [자동] session-start hook — 승격 후보, 미커밋 알림
  → 사용자 첫 메시지
  → [L3 NEW] 즉시 목표(L0) 추출 + recall(키워드)
  → [L3 NEW] L0 + get_context + recall → 세션 목표(L1) 추론
  → 행동 선택: 규칙(L1) + 목표(L3) 기반
```

vs 현재:
```
세션 시작
  → [자동] CLAUDE.md, rules 로드
  → [자동] get_context
  → 사용자 메시지 → 규칙 매칭 → 실행
```

**차이**: recall()을 적극 활용 + 세션 목표 자동 구성.
Pre-flight Recall(S1 구현)이 이것의 첫 단계 — **이미 부분 구현됨**.

### 3-2. 체인 트리거의 목표 기반 전환

현재: 규칙 기반
```yaml
# workflow.md
"구현 완료 감지 → code-reviewer 자동"  # IF-THEN 규칙
```

L3: 목표 기반
```yaml
# 같은 결과, 다른 근거
"구현이 끝났다" → 목표: 품질 보장 → 필요: 코드 리뷰
  → code-reviewer 실행 (규칙이 아니라 목표에서 도출)
```

**실질적 차이**: 규칙에 없는 상황에서도 목표에서 행동을 도출.
예: "타입 마이그레이션 SQL 작성" → 목표: 안전한 마이그레이션 → 필요: 롤백 SQL도 함께 작성.

### 3-3. recall() 품질 = L3 품질

R1 핵심: "recall()의 품질 = 에이전트 자율성의 품질"

NDCG 0.724 → 0.9는 L3 정확도 직결:
- 0.724: "비슷한 텍스트 찾기" — 텍스트 인출
- 0.9: "이 상황에서 필요한 기억 찾기" — 의미적 인출

retrieval_hints가 이 격차를 줄인다:
```
recall("다음 뭐하지")
  → retrieval_hints.when_needed="구현 완료 후 다음 단계 결정 시"
  → 과거 Decision/Pattern 매칭
  → "이전에 비슷한 상황에서 이렇게 했다" 제공
```

### 3-4. 토큰 비용 추정

| 현재 (L2) | L3 추가 | 합계 |
|----------|---------|------|
| get_context: ~200 | recall 1회: ~500 | |
| rules: ~2,000 | 목표 추론: ~300 | |
| **합계: ~2,200** | **+800** | **~3,000** |

세션 당 +800 토큰. 200K 컨텍스트 대비 0.4%. **비용 무시 가능.**

## 4. Phase 0 반영

### 4-1. patch foraging (b-r1-4)

recall() 결과가 한 프로젝트에 편중될 때, 다른 프로젝트로 전환하는 메커니즘.
→ L3에서 활용: "이 프로젝트 내에서 답이 안 나오면 cross-project 검색"

### 4-2. classify→store→link 분리 (a-r1-1)

remember()가 이미 분리된 파이프라인 → L3에서 classify 단계에 목표 맥락 주입 가능.
예: 세션 목표가 "NDCG 개선"이면, remember() 시 자동으로 `project=mcp-memory` + 관련 태그.

## 5. 구현 우선순위

```
P0 (이미 됨): Pre-flight Recall — recall 필수 규칙
P1 (단기):   세션 목표 자동 구성 — dispatch + recall 결합
P2 (중기):   체인 트리거 목표 기반 전환
P3 (장기):   결과 평가 피드백 루프 (세션 끝 달성도 기록)
```
