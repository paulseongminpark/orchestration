# 원칙 — 세션 체인 + 온톨로지 통합

## 1. 기계적 관찰과 해석적 판단은 분리한다

auto_remember(훅)는 기계적이다. 파일명 → 타입, 시그널 → 타입. 판단하지 않는다.
/checkpoint(Claude)는 해석적이다. 대화를 메타적으로 관찰하고 Paul에 대해 이해한다.
이 둘은 대체 불가. 병렬로 돌아가며 서로 다른 정보를 수집한다.

## 2. 세션 데이터는 그래프 노드로 존재한다

sessions 테이블에 JSON으로 묻히면 검색 불가. recall()로 찾을 수 없다.
save_session()은 Narrative(세션 요약) + Decision(결정) + Question(미결)을 개별 노드로 생성하고,
명시적 edge(contains)로 연결한다. link()의 벡터 유사도에 의존하지 않는다.

## 3. 타입은 파일이 결정한다

STATE.md를 수정하면 Decision이다. CLAUDE.md를 수정하면 Principle이다.
파일명과 온톨로지 타입의 매핑은 고정적이고 예측 가능하다.
모호한 것(일반 .md, 코드 파일)은 기록하지 않는다.

## 4. 체인은 짧아야 한다

11단계를 5단계로 줄였다. 각 단계가 정확히 하나의 책임을 갖는다:
LOG → Living Docs → Commit → save_session → Learn.
단계가 많으면 건너뛰기가 생기고, 건너뛰기가 생기면 데이터가 유실된다.

## 5. 비싼 모델은 판단에만 쓴다

compressor는 Sonnet이면 충분하다. 5단계는 실행이지 설계가 아니다.
Opus는 설계·검증·복잡한 실행에 쓴다. 모델 비용 ≠ 품질이 아니라, 모델 비용 = 판단 복잡도.

## 6. 마이그레이션은 추가만 한다

기존 데이터를 삭제하지 않는다. content_hash로 중복을 방어하고, 새 노드를 추가한다.
DRY_RUN을 먼저 돌리고, 확인 후 실행한다. 되돌릴 수 없는 작업은 하지 않는다.

## 7. 모니터링은 구현의 일부다

구현 후 2~3세션 동안 ontology_review로 타입 분포를 확인한다.
Phase 1.5는 "구현 안 하는 Phase"가 아니라, "관찰하는 Phase"다.
문제 발견 시 Phase 1로 돌아간다. 이것이 안전망이다.
