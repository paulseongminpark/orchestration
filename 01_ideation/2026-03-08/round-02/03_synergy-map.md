# 03. 시너지 맵 — 곱이 되는 조합

> 개별로 하면 1+1=2, 같이 하면 1+1=5 되는 조합들.
> "왜 이것들을 같이 해야 하는가"에 대한 근거.

---

## Combo 1: "토큰 해방" (W1 해결)

### 구성: C2 defer_loading + C5 MEMORY.md 분리 + C1 캐시 순서

```
개별 효과:
  C2: -47K 토큰 (MCP 도구)
  C5: -3K 토큰 (MEMORY.md)
  C1: 비용 90% 절감 (캐시 히트)

합산 효과: -50K 토큰 + 캐시 비용 절감

곱셈 효과:
  50K 토큰 회수 → compact 빈도 2~3배 감소
  → compact 감소 → 지식 손실 감소 (W3 부분 해소)
  → 더 긴 세션 → 더 복잡한 작업 가능
  → 세션 격리 문제 완화 (W4 부분 해소)
```

**왜 같이 해야 하나**:
- C2만 하면 47K 회수되지만, MEMORY.md가 여전히 캐시 앞단을 오염
- C5만 하면 3K인데, 도구 55K가 그대로면 의미 미미
- C1(캐시 순서)은 C2와 C5가 **안정된 prefix**를 만들어야 효과 극대화
- 세 개가 맞물려야 "안정 prefix → 높은 캐시 히트율 → 비용+토큰 동시 절감"

**이것 없이 다른 것을 먼저 하면?**:
- W2(Recall) 개선해도 토큰 부족으로 recall 결과를 활용할 context가 없음
- W6(체인 개선)해도 에이전트가 더 많은 context를 먹으면 더 빨리 compact
- **결론: 이것이 1순위. 나머지 모든 개선의 기반.**

## Combo 2: "Recall 혁명" (W2 해결)

### 구성: A1 복합 스코어링 + A4 Half-life + A5 Correction + A6 Multiplier

```
개별 효과:
  A1: similarity 외 2차원 추가 → NDCG +0.05~0.10
  A4: 오래된 노드 자연 감소 → 노이즈 감소
  A5: 교정 top-inject → 반복 실수 제거
  A6: 검증 노드 1.5x → 신뢰 지식 우선

합산 효과: NDCG 0.460 → 0.55~0.65

곱셈 효과:
  더 나은 recall → 더 정확한 컨텍스트 주입
  → 에이전트 판단 품질 향상
  → 더 적은 재작업 → 토큰 절약 (W1 완화)
  → checkpoint 품질 향상 → 지식 손실 감소 (W3 완화)
```

**왜 같이 해야 하나**:
- A1(스코어링)만 넣으면 decay 공식이 없어서 recency 반영 불가
- A4(half-life)만 넣으면 기존 similarity 스코어에 합산할 프레임워크 없음
- A1+A4가 **스코어링 프레임워크**이고, A5+A6은 **오버라이드 레이어**
- A5(correction)는 독립 구현 가능하지만, A1 프레임워크 위에 얹으면 더 정교
- A6(multiplier)는 A1의 importance 축에 자연스럽게 통합

**구현 순서 (내부)**:
```
1. A1 스코어링 프레임워크 (공식 + 가중치)
2. A4 decay 함수 (A1에 plug-in)
3. A6 multiplier (promote_node → 점수 반영)
4. A5 correction (별도 top-inject 로직)
```

## Combo 3: "학습 루프" (W3 + W4 동시 해결)

### 구성: C6 TASK_CONTRACT + B1 Learn + B5 lessons.md + A9 Memory-Merger

```
개별 효과:
  C6: 세션 완료 기준 명확화
  B1: 세션 학습 구조화 추출
  B5: 실패 교훈 누적
  A9: 성숙한 기억 → 규칙 승격

곱셈 효과:
  CONTRACT로 시작 → Learn으로 끝 → lessons에 누적 → Merger로 승격
  = 세션이 "소비"에서 "투자"로 전환

  모든 세션이 시스템을 조금씩 더 똑똑하게 만듦
  → 10세션 후: 에이전트가 과거 10세션의 교훈을 rules로 가지고 있음
  → 20세션 후: 반복 실수 패턴이 자동으로 제거됨
```

**왜 같이 해야 하나**:
- C6(CONTRACT)만 있으면 "뭘 했다"는 알지만 "뭘 배웠다"는 모름
- B1(Learn)만 있으면 배운 건 기록하지만 어디에도 승격 안 됨
- B5(lessons)만 있으면 교훈이 쌓이지만 실행 규칙이 안 됨
- A9(Merger)만 있으면 승격 대상이 없음 (learn이 공급해야)
- **네 개가 파이프라인**: 계약→실행→학습→축적→승격

**이것이 진짜 "인지 시스템"의 핵심**:
- 현재: 저장소 (remember/recall)
- Combo 3 후: 학습하는 시스템 (경험→교훈→규칙→행동 변화)

## Combo 4: "거버넌스 스택" (W7 해결)

### 구성: D1 GovernancePolicy + D3 코드 레벨 + D4 audit hook

```
개별 효과:
  D1: allow/deny/review 정책 정의
  D3: enable_delete=False 등 코드 강제
  D4: 5가지 위협 실시간 스캔

곱셈 효과:
  정책(D1) + 강제(D3) + 감시(D4) = 3중 방어
  → --dangerously-skip-permissions를 지능적으로 대체
  → 안전하게 자율성 확대 (사고 없이 더 많이 위임)
  → 멀티세션 아이디에이션 사고 재발 방지
```

**왜 같이 해야 하나**:
- D1(정책)만 있으면 프롬프트 레벨 → 무시 가능
- D3(코드)만 있으면 경직 (상황별 유연 대응 불가)
- D4(감시)만 있으면 감지만 하고 차단 못 함
- 세 개가 **방어-심층(Defense-in-Depth)**: 정책 선언 → 코드 강제 → 실시간 감시

## Combo 간 상호작용

```
Combo 1 (토큰 해방)
    │
    ├── 기반 제공 ──→ Combo 2 (Recall 혁명)
    │                    │
    │                    └── 품질 향상 ──→ Combo 3 (학습 루프)
    │                                        │
    └── 자율성 확대 ←── Combo 4 (거버넌스) ←──┘
                            │
                            └── 안전하게 더 많이 위임 가능
```

**흐름**:
1. **토큰 해방**이 공간을 만들고
2. **Recall 혁명**이 그 공간에서 더 좋은 정보를 가져오고
3. **학습 루프**가 세션마다 시스템을 성장시키고
4. **거버넌스 스택**이 이 모든 것을 안전하게 자율 운영

이 4개 Combo가 "30개 항목 목록"의 진짜 구조다.
나머지(Wave DAG, gws, GitNexus, LLM Council 등)는 이 4개 기반 위의 확장.
